## 1. Project Overview

> **Mission:** Unite the support and development teams of Inovar Sistemas in a single internal platform — fast, focused, and built around a ninja team culture.

**Project Name:** ShinobiOps  
**Type:** Internal fullstack web application  
**Audience:** Customer Support team and Development team at Inovar Sistemas  

### 1.1 Context & Problem Statement

Inovar Sistemas has two internal teams — Customer Support and Development — that currently lack a structured, real-time communication channel for escalating production issues. When a customer reports a critical bug, the support member must reach developers through informal channels (chat, phone, email), which causes delays, lost context, and poor visibility for the tech lead.

ShinobiOps solves this by providing a purpose-built internal tool where support members open structured tickets and bug reports, developers receive real-time notifications, and the tech lead has full visibility over team load, status, and flow.

### 1.2 Goals

- Reduce time-to-response for critical production bugs reported by customers
- Give the tech lead a real-time overview of every developer's current task and status
- Create a structured, prioritized queue of requests visible and manageable by both teams
- Foster a cohesive, motivated ninja team culture through themed UX
- Operate fully inside the company's internal network with no external dependencies at runtime

### 1.3 Out of Scope (v1.0)

- Email notifications to external parties
- Automatic ClickUp card creation via API (formatted export only)
- Mobile-native application (web responsive only)
- Multi-tenant / multi-company support

## 2. User Roles & Permissions

The system has four roles. Every user self-registers via the registration page and selects their role. Because ShinobiOps runs on an internal network, email validation and admin approval are not required for v1.0.

| Role | Team | Key Privileges |
|------|------|----------------|
| **Tech Lead** | Development | Full dashboard access, manage all settings, assign tickets to anyone, configure checkpoint frequency, manage notification routing, view all devs' status |
| **Developer (Ninja)** | Development | View ticket queue, self-assign tickets, update task status, send help requests, respond to checkpoints |
| **Support Lead** | Support | All Support Member capabilities + can approve/decline any reorder request on behalf of the team |
| **Support Member** | Support | Open tickets and bug reports, request priority reordering, view own ticket/bug history, receive done notifications |

## 3. Authentication & Registration

### 3.1 Registration

A public registration page is available at `/register`. No invitation or admin approval is required.

**Registration fields:**
- Full name (required)
- Email address (required, must be unique)
- Password (required)
- Role — single select: Tech Lead / Developer / Support Member / Support Lead (required)
- Avatar — image upload (optional; system shows initials avatar if none provided)
- Ninja alias — auto-generated by the system, but the user can override it with a custom name (optional). Can also be updated later from profile settings.

### 3.2 Login

Standard email + password login at `/login`. On successful authentication the user is redirected to their role-specific home view. Sessions are persisted via a secure HTTP-only cookie.

### 3.3 Profile

Users can update their name, avatar, and password from their profile settings page. Role changes are not self-service — only the Tech Lead can change a user's role from the admin dashboard.

## 4. Support Team View

### 4.1 Entry Point — Choose Action

After login, support members land on an action selection screen with two prominent options:

- **Report a Bug** — for customer-reported errors that appear to be defects in the software
- **Open a Ticket** — for any other request to the dev team (feature tweak, configuration, data fix, etc.)

---

### 4.2 Open a Ticket

#### Form fields

| Field | Type | Required |
|-------|------|----------|
| Title | Text (max 120 chars) | Yes |
| Description | Rich text / textarea | Yes |
| Severity | Select: Low / Medium / High / Critical | Yes |
| Deadline | Date picker | Yes |
| Attachments | File / image upload | No |

#### Behavior on submit

- Ticket is created with status **Open**
- Inserted into the priority queue — initial position is determined by severity (Critical → top), then by `created_at` (older = higher)
- A sound + visual notification fires on the Development team view

---

### 4.3 Report a Bug

#### Form fields

| Field | Type | Required |
|-------|------|----------|
| Title | Text | Yes |
| Affected module / screen | Text | Yes |
| Steps to reproduce | Numbered list | Yes |
| Expected behavior | Textarea | Yes |
| Actual behavior | Textarea | Yes |
| Severity | Select: Low / Medium / High / Critical | Yes |
| Deadline | Date picker | Yes |
| Environment | Select: Production / Staging / Other | Yes |
| Screenshots / attachments | File upload | No |
| Customer name or ID | Text | No |

#### ClickUp export

A **"Copy to ClickUp format"** button generates a pre-formatted Markdown block the user can paste directly into a ClickUp task description. It includes all fields above in a structured template.

#### Behavior on submit

- Bug is created with status **Open** and flagged as type **Bug** (distinct from Ticket)
- It enters the same shared priority queue
- The dev notification for a Bug is visually and sonically **distinct** from a Ticket notification

---

### 4.4 Priority Queue — Support View

A dedicated page lists all open tickets and bugs, visible to the entire support team. The list is ordered by:

1. Severity (Critical > High > Medium > Low)
2. Then by `created_at` (oldest first within same severity)
3. Manual reorder overrides this default for affected items

#### Reorder request flow

Any support member can request a priority change for a ticket they did not open:

1. Member clicks **"Request reorder"** on a ticket
2. A dialog asks for the desired new position and an optional reason
3. The ticket owner receives an in-app notification with the request and reason
4. The owner can **Approve** or **Decline** the request
5. If approved, the ticket moves to the requested position and all support members see the updated order
6. If declined, the requester receives a notification with the outcome
7. The **Support Lead** can approve or decline any reorder request regardless of ownership. When the Support Lead directly reorders a ticket, the action is marked with a **"Reordered by Lead"** badge and timestamp, visible to the entire support team in the queue.

---

### 4.5 My Tickets & Bugs

Each support member has a personal history page showing all tickets and bugs they have opened, grouped by status: Open, In Progress, Waiting for Info, Done, Cancelled. Each item shows its current status, assignee (if any), and timestamps.

---

### 4.6 Notifications — Support Team

| Event | Notification |
|-------|-------------|
| Ticket / bug marked as Done | In-app + browser notification to the original opener |
| Reorder request received | In-app notification to ticket owner |
| Reorder request approved / declined | In-app notification to requester |
| Ticket status changed | In-app notification to opener |

## 5. Development Team View

### 5.1 Ninja Board — Team Overview

The primary screen for the dev team is the **Ninja Board** — a real-time dashboard showing every developer as a card. Each card displays:

- Developer avatar (or initials if no photo)
- Developer name and ninja alias
- **Current task indicator** — a text box the dev can update at any time
- Current ticket or bug assigned (title + severity badge)
- Status indicator: `Active` / `In Checkpoint` / `Blocked` / `Helping`

---

### 5.2 Ticket Queue — Dev View

The same priority-ordered list visible to support, with additional dev-specific controls:

- **Assign to me** — dev claims a ticket. Assignment is permanent — devs cannot unassign themselves. Only the Tech Lead can reassign a ticket.
- **Assign to another dev** — Tech Lead only
- Filter by type: Ticket | Bug | All
- Filter by severity
- Filter by assignee

---

### 5.3 Ticket Statuses & Transitions

| Status | Description |
|--------|-------------|
| **Open** | Submitted by support, not yet assigned or started |
| **In Progress** | Assigned to a developer actively working on it |
| **Waiting for Info** | Dev needs more information before proceeding |
| **Done** | Resolved — triggers a notification to the support opener |
| **Cancelled** | No longer relevant — triggers a notification to the support opener |

---

### 5.4 Real-Time Notifications — Dev Team

| Event | Notification |
|-------|-------------|
| New Ticket arrives | Sound alert (Tone A) + banner on Ninja Board |
| New Bug arrives | Sound alert (Tone B — distinct from Ticket) + banner on Ninja Board |
| Help request from teammate | Sound alert + pop-up with context message |
| Checkpoint reminder | Sound alert + prompt modal |

> The Tech Lead can configure per-developer whether they receive Ticket notifications, Bug notifications, or both — from the Admin Dashboard.

---

### 5.5 Help Request (Smoke Signal)

Any developer can trigger a help request when they are stuck:

1. Dev clicks the **"Send Smoke Signal"** button (smoke bomb / shuriken icon)
2. A modal prompts for a brief context message — what they are trying to do, what is blocking them (required, max 280 chars)
3. All other developers receive a sound alert + notification banner showing the context
4. A receiving dev can click **"I can help"**, which sets their status to `Helping` and notifies the requester

---

### 5.6 Developer Checkpoint (Status Scroll)

At configurable intervals (set by Tech Lead), the system sends each active developer a checkpoint prompt:

- A sound alert plays and a modal appears
- The dev fills in: current task, whether they are blocked (yes/no), and optional notes
- Submission updates their card on the Ninja Board
- The Tech Lead can view a log of all checkpoint responses per developer
- If a dev does not respond within 10 minutes, their card shows a **"No response"** indicator

---

### 5.7 Developer Task Status Update

At any time, a developer can click their own card on the Ninja Board and update their current task text and status without waiting for a checkpoint. This is the primary "what am I doing right now" signal visible to the whole team.

---

### 5.8 TV Mode — Ninja Board Display

A read-only, full-screen version of the Ninja Board accessible at `/dev/tv`. Designed to be displayed on an office screen without requiring a logged-in session.

**Characteristics:**
- No interactive controls — view only
- Auto-refreshes every 30 seconds (or via SSE for live updates)
- Shows all developer cards with name, alias, avatar, current task, status, and assigned ticket
- Displays a live count of open tickets and bugs by severity at the top
- Minimal chrome — no header, no sidebar, no notifications
- Accessible via a direct URL; no login required (protected only by being on the internal network)
- Tech Lead can enable or disable TV mode access from the Admin Dashboard

## 6. Admin & Tech Lead Dashboard

> Also called: **Command Dojo**

### 6.1 Overview Panel

- Total open tickets and bugs, broken down by severity
- Total tickets assigned vs. unassigned
- Average resolution time (rolling 7-day and 30-day)

### 6.2 Team Management

- View all registered users with role and last active timestamp
- Change a user's role
- Deactivate a user account (soft delete — preserves history)
- Upload or update any user's avatar

### 6.3 Notification Routing

The Tech Lead configures which developers receive which notification types:

| Setting | Scope |
|---------|-------|
| Ticket notifications | Per-developer toggle |
| Bug notifications | Per-developer toggle |
| Help request notifications | Always sent to all devs (not configurable in v1.0) |

### 6.4 Checkpoint Configuration

| Setting | Details |
|---------|---------|
| Enable / disable globally | Boolean toggle |
| Interval | Min 30 minutes, max 8 hours |
| Active hours window | e.g., only fire between 08:00–18:00 |
| Response history | View per-developer log |

### 6.5 Ticket & Bug Log

Full searchable, filterable history of all tickets and bugs ever created, with status, assignee, opener, timestamps, and resolution notes.

### 6.6 TV Mode Settings

- Enable or disable the `/dev/tv` public route
- Configure auto-refresh interval (default: 30 seconds)

## 7. Notifications Specification

### 7.1 In-App Notification Center

All notifications appear in a notification center (bell icon in the header). Unread count is shown as a badge. Clicking a notification navigates to the relevant item.

### 7.2 Sound Alerts

| Event | Sound | Recipients |
|-------|-------|-----------|
| New Ticket | Tone A — subtle chime | Configured dev team members |
| New Bug | Tone B — urgent tone (distinct from A) | Configured dev team members |
| Help Request | Tone C — sharp alert | All devs except requester |
| Checkpoint | Tone D — soft pulse | Individual dev (their own browser) |
| Ticket Done / Cancelled | Tone E — resolution sound | Support opener (their browser) |

Sound plays via the **Web Audio API**. Users can mute their own sounds from profile settings. Sound files are bundled in the application.

## 9. Data Model Overview

> Implementation uses **Prisma ORM** with **MySQL 8.4**.

### User

```
id, name, email, passwordHash
role: TECH_LEAD | DEVELOPER | SUPPORT_LEAD | SUPPORT_MEMBER
avatarUrl (nullable)
ninjaAlias (auto-generated at registration, user-overridable)
isActive, createdAt, updatedAt
notifyTickets (bool), notifyBugs (bool)
```

### Ticket

```
id (internal UUID)
publicId: TKT-XXXX | BUG-XXXX  (human-readable, auto-incrementing per type, immutable)
title, description
type: TICKET | BUG
severity: LOW | MEDIUM | HIGH | CRITICAL
status: OPEN | IN_PROGRESS | WAITING_FOR_INFO | DONE | CANCELLED
deadline, priorityOrder (integer — manual override position)
openedById → User
assignedToId → User (nullable)
createdAt, updatedAt, resolvedAt (nullable)
```

### BugReport (extends Ticket)

```
affectedModule
stepsToReproduce
expectedBehavior
actualBehavior
environment: PRODUCTION | STAGING | OTHER
customerId (nullable)
```

### ReorderRequest

```
id, ticketId → Ticket
requestedById → User
requestedPosition (integer)
reason (nullable)
status: PENDING | APPROVED | DECLINED
resolvedAt, createdAt
```

### HelpRequest

```
id
requestedById → User
contextMessage (max 280 chars)
createdAt
responses: [{ responderId → User, respondedAt }]
```

### Checkpoint

```
id
userId → User
currentTask (text)
isBlocked (bool)
notes (nullable)
createdAt
```

### CheckpointConfig

```
id (single global record)
intervalMinutes
activeHoursStart, activeHoursEnd
isEnabled (bool)
```

### Notification

```
id
userId → User
type (TICKET_NEW | BUG_NEW | HELP_REQUEST | CHECKPOINT | STATUS_CHANGE | REORDER_REQUEST | etc.)
referenceId (polymorphic: ticketId | helpRequestId | etc.)
message
isRead (bool)
createdAt
```

### TicketEvent

```
id
ticketId → Ticket
eventType: CREATED | STATUS_CHANGED | ASSIGNED | REASSIGNED | SEVERITY_CHANGED |
           DEADLINE_CHANGED | PRIORITY_REORDERED | REORDER_REQUESTED |
           REORDER_APPROVED | REORDER_DECLINED | HELP_REQUESTED |
           CHECKPOINT_MENTION | DONE | CANCELLED
actorId → User
metadata (JSON — stores old/new values relevant to the event type)
createdAt (immutable — never updated)
```

## 10. Ticket & Bug Timeline

Every ticket and bug has a unique human-readable ID and a full, immutable timeline of every event that occurred during its lifecycle. This provides complete traceability for stakeholders, the tech lead, or any team member who needs to understand exactly what happened with a given item.

### 10.1 Ticket ID Format

Each ticket and bug receives a short, readable ID generated at creation time:

| Type | Format | Example |
|------|--------|---------|
| Ticket | `TKT-XXXX` | `TKT-0042` |
| Bug | `BUG-XXXX` | `BUG-0017` |

The numeric part is a zero-padded auto-incrementing sequence per type. IDs are **permanent and never reused**.

### 10.2 Timeline Events

Every state change, assignment, or notable action is recorded as an immutable `TicketEvent` entry. The following events are tracked:

| Event Type | Description |
|------------|-------------|
| `CREATED` | Ticket or bug was opened — records opener and all initial field values |
| `STATUS_CHANGED` | Status transition (e.g., Open → In Progress) — records old and new status, actor |
| `ASSIGNED` | Ticket assigned to a developer — records assignee name and actor |
| `REASSIGNED` | Tech Lead changed the assignee — records old and new assignee, actor |
| `SEVERITY_CHANGED` | Severity updated — records old and new severity, actor |
| `DEADLINE_CHANGED` | Deadline updated — records old and new date, actor |
| `PRIORITY_REORDERED` | Position in queue changed — records old/new position, actor, whether it was a Lead override or an approved request |
| `REORDER_REQUESTED` | A reorder request was submitted — records requester, target position, reason |
| `REORDER_APPROVED` | Reorder request approved by owner or Support Lead |
| `REORDER_DECLINED` | Reorder request declined — records reason if provided |
| `HELP_REQUESTED` | A developer sent a Smoke Signal referencing this ticket |
| `CHECKPOINT_MENTION` | A developer mentioned this ticket in a Status Scroll checkpoint response |
| `DONE` | Ticket marked as resolved — records resolver and timestamp |
| `CANCELLED` | Ticket cancelled — records actor and timestamp |

### 10.3 Ticket Detail & Timeline Page

Each ticket and bug has a dedicated detail page accessible by its public ID.

**URL pattern:**
```
/ticket/TKT-0042
/ticket/BUG-0017
```

**Page content:**

- **Header** — public ID (copyable), title, type badge, severity badge, current status, opener name + avatar, assignee name + avatar, created date, deadline
- **Original fields** — full description; for bugs: affected module, steps to reproduce, expected vs. actual behavior, environment, customer ID
- **ClickUp export button** — visible on bug-type items only
- **Timeline feed** — chronological list of all events, each entry showing:
  - Event icon and human-readable label (e.g., *"Assigned to Ryu by Jōnin Tanaka"*)
  - Actor name and avatar
  - Exact date and time
- **Direct shareable link** — authentication required, but any logged-in user on the internal network can access any ticket by URL

### 10.4 ID Visibility & Search

- The public ID (`TKT-XXXX` / `BUG-XXXX`) is displayed on:
  - Every card in the Mission Board queue
  - The assigned-ticket field on each developer card on the Ninja Board
  - All in-app notification messages
  - The TV Mode Ninja Board
- The ID is **copyable with a single click** anywhere it appears
- The global search bar (available in the header for all roles) accepts a raw ID and navigates directly to the detail page

## 13. Non-Functional Requirements

### 13.1 Performance

- Page load under 2 seconds on a LAN connection
- Real-time notifications delivered within 1 second of the triggering event
- MySQL InnoDB engine ensures concurrent read/write performance

### 13.2 Reliability

- MySQL data stored on a Docker named volume — data survives container restarts and upgrades
- Graceful error handling on all API routes with user-friendly error messages

### 13.3 Security

- Passwords hashed with bcrypt (cost factor ≥ 12)
- All authenticated routes require a valid session cookie
- Role-based access checks enforced on every API route (not just the UI)
- MySQL only accessible within the Docker network

### 13.4 Usability

- Self-explanatory UI — fully usable without documentation
- Responsive layout — usable at 1280px desktop and tablet sizes
- Sound alerts can be muted per user from profile settings

## 14. Resolved Decisions

| Question | Decision |
|----------|----------|
| Should developers be able to reject / unassign a ticket after claiming it? | **No.** Once a developer assigns a ticket to themselves, the assignment is permanent. Only the Tech Lead can reassign it. |
| Should there be a comment thread on each ticket? | **No.** Not in v1.0. |
| Ninja alias — auto-generated or user-chosen at registration? | **Both.** The registration form auto-generates a ninja alias, but the user can override it with a custom one. It can also be updated later from profile settings. |
| Should the Support Lead have a distinct queue view from Support Member? | **No distinct view.** Same queue page, but Support Lead reorder actions are visually distinguished in the queue with a "Reordered by Lead" badge and a timestamp, visible to the entire support team. |
| Should there be a read-only "TV mode" for the Ninja Board? | **Yes.** See Section 5.8 below. |
